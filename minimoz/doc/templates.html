Minimoz Templates

The general concept consists of building blocks and pages. A building block is a functional element which is included in a page. Each building block can be defined multiple times for different client nls environments to support content selection depending on the clients nls preferences. Note that no building block or page can exceed 8192 bytes in length in the template file, which, however, should never be any problem at all (there is always the var building block if long page content is required).

Building Blocks

Any building block has the following form:

<+bbspecifier:name[:nlslist]>bbdata<-bbspecifier>

  • bbspecifier: the specifier that actually states the kind of building block defined here. See below for the types of building blocks available. Note that the clear, depclear, save, restore and csslink building blocks cannot be defined.
  • name: the name of this building block. The name needs not to be unique as the same building block may be defined more than once with different nls contents. Note that the webnews and searchengine building blocks cannot have a name or a nlslist specifier.
  • nlslist: an optional blank separated list of nls support identifiers for which this building block is valid, e.g. "en en_us de de_de".
  • bbdata: the data associated with this building block. This is, depending on the type of the building block, either an immediate data string or one or more options which are specified as: <+option>optiondata<-option> where option is the option identifier and optiondata is the data string associated with the option. The option opts always contains a blank separated list of options for the building block.

The final columns of the above table describes the validity of a building block for the available page types:
  • C  -  catalog page including home and index page
  • I    -  info page
  • A  -  advanced search page
  • Q  -  query output page
  • S   -  css (cascading style sheet) page
  • R  -  robots page
  • E  -  error page
An x means the building block can be used for the page type, o means that the building block can only be used for the http 302 redirect page.

Additional Information about Images

Minimoz preloads images and uses the image building block name for image references. Gif images may be referenced by /img/name.gif, jpeg images by /img/name.jpg and png images by /img/name.png where name is the image building block name. It is important to know how images are addressed as building blocks can't include other building blocks which means img tags in building blocks have to be defined manually. Keep in mind that you cannot address images not preloaded by minimoz.

Pages

Pages define the actual output elements that can be retrieved by minimoz clients. Any page is specified using the following form:

<+pagename>page data<-pagename>

The pagename is one of the names defined below. Any page data consists of data sent directly to the client and inserted building blocks. Any building block is inserted in a page by including the following form (the building block must be defined before it can be inserted):

<@bbspecifier:name>

Note that there is no name for the clear, depclear, save, restore and csslink building blocks.

NLS Support

As you can see above the nlslist of the building blocks defined before are not included when a building block is referenced in a page. This selection is done automatically by evaluation of the Accept-Language header transmitted by the user agent (browser). Thus it is possible to define building blocks in various languages and to have them output to the client depending on the clients language preferences. The selection is as follows between all building blocks of the same type with the same name:
  1. Use the building block that defines the nls identifier which is most left in the Accept-Language header sent by the client (to be flexible the comparison is case insensitive with '-' being mapped to '_')
  2. Use the first building block that has no nlslist specified
  3. Use the last building block specified

Conditional Processing

There are some building blocks for conditional processing. Their inclusion in a page allows conditional output depending on output flag values. There are two flag values:
  1. output flag value
  2. dependent output flag value (only evaluated if the output flag is true)
Both values are simultaneously set by building blocks being processed. Building blocks setting the values are text oriented building blocks that can fail to output regular data either due to database data being not available or due to database failures. If these building blocks do database oriented output they do set both output flags.

Furthermore both output flags can be saved and restored simultaneously and they can be cleared independently of each other. Then the true, deptrue, false and depfalse conditionals can be used to output data to the client according to the state of the output flags.

You can use the conditional processing e.g. to output some text, if there can neither category nor external page data be found in the database for the keywords given by a user. Keep in mind that the conditional processing can be combined with nls support, too.

The page you will have to begin with when you define your own template is the catalogpage. It is the most common page which is used to display any of the available categories. Then you probably will want to define the infopage to be able to access the category descriptions as well as the querypage to be able to search the database.

Having set up the basic pages you probably want to define a homepage with some custom layout, probably in your preferred language(s). To be able to either access the english home page or the top level catalog in standard format you then can define an indexpage for this purpose.

Additionally you can define the advancedpage if you want to make the advanced search form available and you can define the robotspage for crawler exclusion.

Finally you can define custom error pages in the case you either want to have translated error messages or a custom error page layout.
